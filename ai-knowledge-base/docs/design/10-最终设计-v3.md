# AI Knowledge Base — 最终收敛设计 v3.0

> 最终版，作为开发唯一依据
> 定位：最小可运行 + 可扩展骨架，不砍 RAG 能力，不砍 SSE

---

## 一、设计原则

| 原则 | 说明 |
|------|------|
| **先跑通** | 所有代码必须能在 1 周内写完可运行版本 |
| **不破坏扩展** | chunk 表独立存在，VectorStore 接口抽象，后续加 Milvus 零改动 |
| **砍包装不砍内核** | 砍 Security 框架 / 前端 / Docker / Admin，不砍 RAG / SSE / Chunk |
| **扁平目录** | 不分 module 子包，controller/service/entity 三层直排 |

## 二、最终数据库设计（4 张表，稳定版）

```sql
CREATE DATABASE IF NOT EXISTS ai_knowledge_base DEFAULT CHARSET utf8mb4;
USE ai_knowledge_base;

-- ============================================
-- 1. 用户表
-- ============================================
CREATE TABLE `user` (
    `id`           BIGINT       PRIMARY KEY AUTO_INCREMENT,
    `username`     VARCHAR(50)  NOT NULL UNIQUE,
    `password`     VARCHAR(255) NOT NULL COMMENT 'BCrypt',
    `create_time`  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `update_time`  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- ============================================
-- 2. 知识库表
-- ============================================
CREATE TABLE `knowledge` (
    `id`           BIGINT        PRIMARY KEY AUTO_INCREMENT,
    `user_id`      BIGINT        NOT NULL,
    `name`         VARCHAR(100)  NOT NULL,
    `description`  VARCHAR(500)  DEFAULT NULL,
    `create_time`  DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `update_time`  DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX `idx_user_id` (`user_id`)
);

-- ============================================
-- 3. 文档表
-- ============================================
CREATE TABLE `document` (
    `id`            BIGINT        PRIMARY KEY AUTO_INCREMENT,
    `knowledge_id`  BIGINT        NOT NULL,
    `file_name`     VARCHAR(255)  NOT NULL,
    `file_type`     VARCHAR(20)   NOT NULL,
    `file_size`     BIGINT        NOT NULL,
    `file_path`     VARCHAR(500)  NOT NULL,
    `chunk_count`   INT           NOT NULL DEFAULT 0,
    `status`        TINYINT       NOT NULL DEFAULT 0 COMMENT '0待处理 1处理中 2已完成 3失败',
    `error_msg`     VARCHAR(500)  DEFAULT NULL,
    `create_time`   DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `update_time`   DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX `idx_knowledge_id` (`knowledge_id`)
);

-- ============================================
-- 4. 文档分块表（RAG 核心，必须保留）
-- ============================================
CREATE TABLE `document_chunk` (
    `id`            BIGINT        PRIMARY KEY AUTO_INCREMENT,
    `document_id`   BIGINT        NOT NULL,
    `knowledge_id`  BIGINT        NOT NULL COMMENT '冗余，便于检索和过滤',
    `chunk_index`   INT           NOT NULL COMMENT '分块序号',
    `content`       TEXT          NOT NULL COMMENT '分块原文',
    `vector_id`     VARCHAR(100)  DEFAULT NULL COMMENT '向量存储中的ID，关联 VectorStore',
    `create_time`   DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX `idx_document_id` (`document_id`),
    INDEX `idx_knowledge_id` (`knowledge_id`)
);
```

### 表关系

```
user 1:N knowledge 1:N document 1:N document_chunk
                                        │
                                        └── vector_id ──▶ VectorStore（抽象）
```

### 为什么不砍 chunk 表

| 场景 | 需要 chunk 表 |
|------|:---:|
| RAG 引用溯源（显示原文来源） | ✅ |
| 按 knowledge_id 过滤向量检索范围 | ✅ |
| 删除文档时清理向量（知道 vector_id） | ✅ |
| 后续升级 Milvus，仍需 MySQL 存原文 | ✅ |
| 查询某个知识库有多少分块 | ✅ |

**chunk 表是 RAG 的锚点，砍掉后向量数据变成孤儿，无法溯源也无法清理。**

## 五、核心流程设计

### 5.1 文档处理流程（异步）

```
DocumentController 上传
    │
    ├── 同步：存文件 + 插入 document(status=0) → 返回
    │
    └── @Async DocumentService.processDocument(documentId)
        ├── status → 1 (处理中)
        ├── Tika 解析 → 纯文本
        ├── 分块（按 \n\n 切，每块 ≤ 500 字，重叠 50 字）
        ├── 每块:
        │   ├── 调 EmbeddingService.embed(chunkText) → float[]
        │   ├── 写入 VectorStore（metadata: documentId, knowledgeId, fileName, chunkIndex）
        │   └── 插入 document_chunk 表（记录 vector_id）
        ├── 更新 document.chunk_count
        └── status → 2 (完成) 或 status → 3 (失败)
```

### 5.2 RAG 对话流程（SSE）

```
ChatController.chat(request)
    │
    ├── knowledgeId == null → 普通对话
    │   └── ChatClient.prompt(message).stream() → Flux<SseEvent>
    │
    └── knowledgeId != null → RAG 对话
        ├── RagService.retrieve(message, knowledgeId, topK=3)
        │   ├── EmbeddingService.embed(message) → queryVector
        │   ├── VectorStore.similaritySearch(queryVector, topK, filter=knowledgeId)
        │   └── 返回 List<ChunkResult>（含原文 + 分数）
        ├── 组装 Prompt:
        │     System: rag-system.txt 内容
        │     Context: 检索到的 3 段原文
        │     User: message
        ├── ChatClient.prompt(prompt).stream() → Flux<SseEvent>
        └── 流结束后发送 done 事件（附带 references）
```

### 5.3 SSE 事件格式

```
event: message
data: {"content":"逐字输出的文本片段"}

event: done
data: {"references":[{"content":"原文片段","fileName":"guide.pdf","score":0.92}]}

event: error
data: {"msg":"AI模型调用失败"}
```

## 九、扩展路径（不破坏现有结构）

```
当前 MVP 结构                        后续扩展（零改动现有代码）
───────────────────────────────────────────────────────────────
SimpleVectorStore                →   Milvus（换 AiConfig 一行）
JSON 返回对话                    →   SSE（换 Controller 返回类型）
不持久化对话历史                  →   加 chat_session + chat_message 表
手写 JwtFilter                   →   换 Spring Security（Filter 兼容）
单 application.yml               →   加 application-dev.yml / prod.yml
无前端                           →   加 Vue 3 前端项目
物理删除                         →   加 deleted 字段 + @TableLogic
本地 ./uploads                   →   换 MinIO
```

**所有扩展都是"加新代码"或"换配置"，不需要重写现有逻辑。**
