## 3. 模块架构

### 3.1 模块依赖关系

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  user    │     │knowledge │     │   ai     │     │  admin   │
│  模块    │     │  模块    │     │  模块    │     │  模块    │
└────┬─────┘     └────┬─────┘     └────┬─────┘     └────┬─────┘
     │                │                │                │
     │                │                │                │
     ▼                ▼                ▼                ▼
┌────────────────────────────────────────────────────────────┐
│                      common 公共层                          │
│  R<T> │ Exception │ JWT │ Security │ Redis │ Constants     │
└────────────────────────────────────────────────────────────┘
     │
     ▼
┌────────────────────────────────────────────────────────────┐
│                    基础设施层                                │
│  MySQL │ Redis │ Spring AI │ Tika │ 文件存储               │
└────────────────────────────────────────────────────────────┘
```

**模块间关系说明：**

| 调用方 | 被调用方 | 场景 |
|--------|---------|------|
| ai → knowledge | RAG 对话需要查询知识库信息 |
| ai → common | 使用 JWT 获取当前用户 |
| knowledge → ai | 文档入库时调用 Embedding |
| admin → user | 管理员操作用户数据 |
| 所有模块 → common | 统一响应、异常、鉴权 |

### 3.2 各模块职责

| 模块 | 职责 | 核心类 |
|------|------|--------|
| **user** | 用户注册、登录、JWT、个人信息 | UserService, UserController |
| **knowledge** | 知识库 CRUD、文档上传/解析/分块 | KnowledgeService, DocumentService |
| **ai** | 对话管理、RAG 检索、Embedding、SSE 流式 | ChatService, RagService, EmbeddingService |
| **admin** | 仪表盘、用户管理、系统配置 | DashboardService, AdminController |
| **common** | 统一响应、异常处理、JWT 工具、常量 | R, GlobalExceptionHandler, JwtUtil |

## 5. 安全架构

### 5.1 认证鉴权流程

```
┌────────┐    ① 登录(用户名+密码)    ┌────────────┐
│  客户端  │ ──────────────────────▶ │ AuthController│
│        │ ◀──────────────────────  │             │
└────────┘    ② 返回 JWT Token       └────────────┘
     │
     │ ③ 请求 API (Header: Authorization: Bearer <token>)
     ▼
┌────────────┐   ④ JWT 过滤器验证    ┌────────────┐
│  Filter    │ ────────────────────▶│ Security   │
│ JwtFilter  │   ⑤ 解析 userId      │ Context    │
└────────────┘   ⑥ 放行/拒绝        └────────────┘
```

### 5.2 数据隔离策略

```
所有业务查询都必须带 user_id 过滤条件:

SELECT * FROM knowledge WHERE user_id = #{currentUserId} AND deleted = 0

确保用户无法访问其他用户的数据。
知识库可选择性开放为"公开"，公开库所有人可读。
```

## 6. 缓存架构

### 6.1 Redis 使用场景

| Key 模式 | 用途 | TTL |
|---------|------|-----|
| `user:token:{userId}` | JWT Token 白名单 | 与 Token 有效期一致 |
| `chat:session:{sessionId}` | 最近 N 轮对话上下文 | 30 分钟（每次对话刷新） |
| `chat:rate:{userId}` | 对话频率限制 | 1 分钟滑动窗口 |
| `kb:stats:{knowledgeId}` | 知识库统计数据缓存 | 10 分钟 |

## 7. 可扩展性设计

### 7.1 AI 模型可插拔

```java
// 通过配置文件切换模型，代码无需修改
spring.ai.openai.api-key=${AI_API_KEY}
spring.ai.openai.base-url=${AI_BASE_URL}   // OpenAI / Ollama / 通义千问
spring.ai.openai.chat.options.model=${AI_MODEL}
```

### 7.2 向量存储可升级

```
Phase 1: SimpleVectorStore          ← 开发验证，零依赖
    │
    ▼ 配置切换，接口不变
Phase 2: PGVector                   ← 生产可用，PostgreSQL 插件
    │
    ▼ 配置切换，接口不变
Phase 3: Milvus                     ← 大规模生产，独立向量数据库
```

Spring AI 的 `VectorStore` 接口抽象了底层实现，切换只需更换依赖和配置。

### 7.3 文档解析器可扩展

```
Parser 接口（策略模式）:
├── PdfParser      ─ Tika 解析 PDF
├── DocxParser     ─ Tika 解析 Word
├── TextParser     ─ 纯文本读取
├── MarkdownParser ─ Markdown 结构化解析
├── CsvParser      ─ CSV 表格解析
└── ... 未来可扩展 Excel, PPT 等
```
